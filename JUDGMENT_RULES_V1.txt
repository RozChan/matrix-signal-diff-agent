matrix-signal-diff-agent 1.0 判断规则说明（必须随规则修改同步更新）
====================================================================

文件用途
--------
本文件用于给业务、测试、后续开发人员查看当前 1.0 版本 legacy 脚本中的全部自动判断规则。
本文件选择 TXT 格式，因为：
1. 不依赖 Word / Excel 软件；
2. Git diff 可直接审阅；
3. 适合随代码版本一起维护；
4. 后续规则变更时容易同步修改。

维护要求
--------
如果以后修改以下任一脚本中的判断规则，必须同步更新本文件：
- common_matrix_utils_local_v13.py
- 01_extract_full_matrix_local_v13.py
- 02_generate_dedup_signals_local_v13.py
- 03_generate_compare_file5_local_v13.py

本文件只描述当前脚本已有规则，不代表推荐规则，也不代表未来增强方向。


一、整体流程规则
================

1. 输入目录规则
---------------
01_extract_full_matrix_local_v13.py 从脚本所在目录下读取：
- input/4.0
- input/5.1

Streamlit Demo 的 core/pipeline.py 会把 legacy 脚本复制到每次任务目录，并以任务目录作为 cwd 执行，因此 legacy 脚本实际读取的是：
- temp/<task_id>/input/4.0
- temp/<task_id>/input/5.1

2. 输入文件类型规则
-------------------
只处理满足以下条件的 Excel 文件：
- 后缀是 .xlsx 或 .xlsm；
- 是文件，不是目录；
- 文件名不以 ~$ 开头。

不支持：
- .xls
- .csv
- .dbc
- .arxml
- 压缩包
- 加密或损坏 Excel

3. 三阶段输出规则
-----------------
阶段 01：全量提取
- output/26R1 4.0全量信号矩阵清单.xlsx
- output/26R2 5.1全量信号矩阵清单.xlsx

阶段 02：同名去重
- output/26R1 4.0全量信号-同名去重后.xlsx
- output/26R2 5.1全量信号-同名去重后.xlsx

阶段 03：最终差异识别
- output/4.0和5.1同一信号差异点识别.xlsx

4. 异常处理规则
---------------
- 01 脚本：单个 workbook 读取失败时记录日志并跳过该文件；单个 sheet 处理失败时记录日志并跳过该 sheet。
- 02 / 03 脚本：关键输入文件或关键列缺失会抛异常，脚本退出码为 1。
- Streamlit pipeline：任何 legacy 脚本返回非 0 退出码时，抛出包含脚本名、返回码、stdout、stderr 的异常。


二、公共文本清洗规则
====================

1. cell_text(value)
-------------------
所有主要字段读取时会经过 cell_text：
- None -> 空字符串 ""；
- 转成 str；
- 全角空格 \u3000 -> 普通空格；
- \r\n 和 \r -> \n；
- 去掉首尾空白。

影响：
- 首尾空格不参与判断；
- Windows / Unix 换行会统一；
- 全角空格会先转普通空格。

风险：
- 中间空格是否保留取决于后续字段比较逻辑；
- Excel 原始数字格式可能已经被 openpyxl 读成 Python 数值。

2. norm_header(value)
---------------------
表头识别用 norm_header：
- 先 cell_text；
- 转小写；
- 删除所有非数字、非英文字母、非中文字符。

影响：
- 表头大小写差异忽略；
- 表头中的空格、下划线、括号、标点差异忽略；
- 中文和英文字符保留。

风险：
- 过度归一化后，不同表头可能被识别成相似文本；
- 只用于表头匹配，不用于最终字段值比较。


三、01 全量提取规则
===================

1. 矩阵 sheet 自动识别规则
-------------------------
脚本不会直接按 sheet 名过滤，而是按表头规则判断是否是真矩阵 sheet。

扫描范围：
- 每个 sheet 最多扫描前 40 行作为候选表头行。

核心字段列表：
- msg_id
- frame_format
- msg_name
- msg_length
- msg_send_type
- transmitter
- signal_name
- start_bit
- bit_length
- data_type
- resolution
- offset
- signal_min
- signal_max

必须存在字段：
- signal_name
- bit_length
- resolution
- offset
- signal_min
- signal_max

允许缺失规则：
- 上述核心字段最多允许缺失 1 个；
- 但必须存在字段一个都不能缺。

候选行选择规则：
- 在前 40 行中，所有通过校验的候选表头行里，选择命中核心字段数量最多的那一行。

结果：
- History、修改记录、说明页等非矩阵 sheet 通常因为无法通过核心表头校验而被跳过。

风险：
- 如果非矩阵 sheet 偶然包含足够多核心表头，可能被误收；
- 如果真实矩阵 sheet 表头命名超出别名列表，可能被跳过；
- 只扫描前 40 行，表头在第 41 行以后会被跳过。

3. 表头匹配规则
---------------
- 读取单元格文本后 norm_header；
- 读取别名后 norm_header；
- 如果表头归一化文本 == 别名归一化文本，匹配成功；
- 如果别名归一化文本包含在表头归一化文本里，匹配成功。

风险：
- 包含匹配可能带来误识别；
- 例如某些长表头中包含短别名时可能被认为是目标字段。

4. 信号行名称规则
-----------------
信号名称为空时，不认为是信号行。

以下归一化后的名称不认为是信号行：
- signalname
- signal
- signalname信号名称
- 信号名称
- 信号名

其余非空名称视为候选信号名称。

5. 信号定义行判断规则
---------------------
一行是否真正作为信号记录输出，需要满足：
- signal_name 是有效信号行名称；
- 在以下证据字段中至少有 2 个非空：
  - start_bit
  - bit_length
  - resolution
  - offset
  - signal_min
  - signal_max

如果不满足，会被当作非信号定义行，用于更新当前报文级上下文。

风险：
- 某些信号行如果证据字段缺失过多，会被跳过；
- 某些非信号行如果误填了足够证据字段，可能被误收。

6. 报文级字段继承规则
---------------------
当遇到非信号定义行时，脚本会从该行读取并缓存以下报文级字段：
- msg_id
- frame_format
- msg_name
- msg_length
- msg_send_type
- transmitter

当后续信号定义行这些字段为空时，会继承最近缓存的报文级字段。

风险：
- 如果矩阵中报文块结构异常，可能继承到错误上下文。

7. ECU 状态列识别规则
---------------------
ECU 状态列只从“信号定义字段右侧”开始扫描：
- right_start_col = max(field_map.values()) + 1

用于判断 ECU 列的信号行样本：
- 表头行之后的候选信号行；
- 最多取前 1000 个候选信号行。

列排除关键词：
如果表头归一化文本包含以下关键词，则不作为 ECU 状态列：
- asil
- safety
- 安全
- comment
- remark
- 备注
- description
- 描述
- history
- version
- 版本
- date
- time
- 修改
- issue

状态列成立条件：
- 该列至少有 1 个状态值；
- 在非空样本中，状态值占比必须 >= 60%。

状态值定义：
- R/r -> r
- T/t/S/s -> s
- X/x/× -> x
- 其他值不算状态值

风险：
- ECU 状态列在信号定义字段左侧时识别不到；
- 样本中状态值填得太少，低于 60% 会被跳过；
- 只看最多 1000 个候选信号行；
- 某些非 ECU 列如果大量出现 R/T/S/X，也可能误识别。

8. ECU 状态标准化规则
---------------------
normalize_ecu_state：
- 空值 -> 空字符串
- R/r -> r，表示接收 receive
- T/t/S/s -> s，表示发送 send
- X/x/× -> x，表示不参与
- 其他 -> 空字符串

注意：
- x 不参与输出；
- 只有 r/s 会进入发送或接收汇总。

9. 4.0 ECU 状态输出格式
-----------------------
4.0 架构下，ECU 状态输出采用“第一个 ECU 为主，后续 ECU 放括号”的格式。

原始状态示例：
EMS_PHEV:S(HighRegulationArea:S,LowRegulationArea:S)

标准化状态示例：
EMS_PHEV:s(HighRegulationArea:s,LowRegulationArea:s)

发送 ECU 汇总 / 接收 ECU 汇总也采用括号格式：
EMS_PHEV(HighRegulationArea,LowRegulationArea)

规则：
- 按 ECU 列从左到右顺序；
- 第一个非空 r/s 状态 ECU 为主；
- 后续非空 r/s 状态 ECU 放括号；
- x、空值、未知状态不输出。

10. 5.1 ECU 状态输出格式
------------------------
5.1 架构下，不做括号合并，保持逐行输出：
VCU_5IC_DK:s
VCU_5IH_DK:s

发送 ECU 汇总 / 接收 ECU 汇总使用换行连接，并保持首次出现顺序去重。

四、02 同名去重规则
===================

1. 输入文件规则
---------------
读取 01 阶段输出文件：
- output/26R1 4.0全量信号矩阵清单.xlsx
- output/26R2 5.1全量信号矩阵清单.xlsx

2. 列识别规则
-------------
02 脚本最多扫描前 10 行寻找表头。

必须识别到 signal_name，否则抛异常。

列匹配方式：
- cell_text；
- norm_header；
- 表头归一化后等于候选名归一化文本，或包含候选名归一化文本。

3. 去重依据
-----------
按 signal_name 去重。

规则：
- signal_name 经过 cell_text；
- 空信号名跳过；
- 分组 key 是 cell_text 后的原始大小写字符串；
- 大小写不同的信号名会被视为不同信号；
- 中间空格、下划线、短横线不同也会被视为不同信号。

4. 去重保留规则
---------------
同一 signal_name 分组内：
- 信号定义字段保留首次出现记录；
- 后续出现的同名信号不会覆盖定义字段。

保留首次出现的字段包括：
- 信号名称
- 信号长度
- 精度
- 偏移量
- 物理最小值
- 物理最大值
- 单位
- 信号值描述

风险：
- 如果首次出现记录不是最准确记录，后续更完整定义不会覆盖。

5. 来源文件汇总规则
-------------------
信号来源文件汇总所有出现过该信号的 source_file：
- 去空；
- 保持首次出现顺序；
- 去重；
- 用换行连接。

6. ECU 状态汇总规则
-------------------
以下字段直接按文本行汇总并去重：
- ECU收发状态_原始
- ECU收发状态_标准化
- 发送ECU汇总
- 接收ECU汇总

规则：
- cell_text；
- 按换行拆行；
- 去掉空行；
- 保持首次出现顺序；
- 去重；
- 用换行连接。

特别规则：
- 4.0 的括号格式不拆散；
- EMS_PHEV:s(HighRegulationArea:s,LowRegulationArea:s) 会作为一整行保留。

7. 去重输出字段
---------------
02 输出固定列：
- 信号名称
- 信号长度
- 精度
- 偏移量
- 物理最小值
- 物理最大值
- 单位
- 信号值描述
- 信号来源文件
- ECU收发状态_原始
- ECU收发状态_标准化
- 发送ECU汇总
- 接收ECU汇总


五、03 最终差异识别规则
=======================

1. 输入文件规则
---------------
读取 02 阶段输出文件：
- output/26R1 4.0全量信号-同名去重后.xlsx
- output/26R2 5.1全量信号-同名去重后.xlsx

2. 列识别规则
-------------
03 脚本最多扫描前 10 行寻找表头。

必须识别到 signal_name，否则抛异常。

列匹配方式同 02：
- cell_text；
- norm_header；
- 表头归一化后等于候选名归一化文本，或包含候选名归一化文本。

3. 最终参与比较的 7 个字段
--------------------------
只有以下字段参与最终差异筛选：
- 信号长度 bit_length
- 精度 resolution
- 偏移量 offset
- 物理最小值 signal_min
- 物理最大值 signal_max
- 单位 unit
- 信号值描述 signal_value_description

4. 不参与最终差异筛选的字段
---------------------------
以下字段只作为追溯字段带出，不参与差异筛选：
- 信号来源文件
- ECU收发状态_原始
- ECU收发状态_标准化
- 发送ECU汇总
- 接收ECU汇总

以下全量阶段存在的字段也不参与 03 最终差异判断：
- Msg ID
- Frame Format
- Msg Name
- Msg Length(Byte)
- Msg Send Type
- Transmitter
- Start Bit
- Data Type
- 备注
- 来源文件
- 来源Sheet
- 来源行号

因此当前 03 不输出以下差异：
- 新增信号
- 删除信号
- Start Bit 差异
- Data Type 差异
- 报文 ID 差异
- 报文名差异
- Transmitter 差异
- ECU 发送/接收变化

5. sheet1：完全同名匹配规则
--------------------------
sheet1 名称：完全同名匹配对比结果。

匹配规则：
- 取 4.0 去重信号名集合和 5.1 去重信号名集合的交集；
- signal_name 只经过 cell_text，不做大小写归一；
- 大小写不同不算同名；
- 中间空格、下划线、短横线不同不算同名；
- 首尾空白会被 cell_text 去掉。

输出规则：
- 匹配成功后，比较 7 个字段；
- 只有至少 1 个字段存在差异时才输出该行；
- 如果 7 个字段都一致，不输出。

6. sheet2：vcu-hcu 同名匹配规则
------------------------------
sheet2 名称：vcu-hcu 同名匹配。

候选规则：
- 只处理 sheet1 未匹配的剩余信号；
- 4.0 和 5.1 信号名都必须以 VCU 或 HCU 开头；
- 判断是否以 VCU/HCU 开头时忽略大小写；
- 去掉开头 VCU 或 HCU，以及其后可选的 _ 或 -；
- 用去前缀后的剩余字符串作为匹配 key。

示例：
- VCU_Speed -> Speed
- HCU_Speed -> Speed
- VCU-Speed -> Speed
- HCUSpeed -> Speed

注意：
- 去前缀后的剩余 key 没有统一转小写；
- 因此 Speed 与 speed 不匹配；
- 如果同一 key 有多个 5.1 候选，取第一个未使用候选；
- 每个 5.1 候选最多使用一次。

输出规则：
- 匹配成功后，比较 7 个字段；
- 只有至少 1 个字段存在差异时才输出该行；
- 如果 7 个字段都一致，不输出。

7. 差异点 list 生成规则
----------------------
对 7 个字段逐项比较。

若字段不一致，差异点 list 添加一行：
字段中文名：4.0=<4.0值>；5.1=<5.1值>

多个字段差异用换行连接。


六、字段值相等判断规则 values_equal
==================================

1. 数值字段规则
---------------
适用字段：
- bit_length
- resolution
- offset
- signal_min
- signal_max

比较流程：
1. 对两个值分别 cell_text；
2. 删除英文逗号 ,；
3. 尝试转 Decimal；
4. 如果两个值都能转 Decimal，则直接比较 Decimal 是否相等；
5. 如果任一值不能转 Decimal，则走普通文本比较规则。

可自动认为一致的例子：
- 1 与 1.0
- 1.00 与 1
- 1,000 与 1000
- 1,000.0 与 1000
- Decimal 可解析的科学计数法与普通小数

风险：
- 如果显示精度本身有业务意义，1 与 1.0 会被忽略；
- 如果逗号不是千分位，也会被删除；
- 带单位的数值无法按数值比较，会退回文本比较。

2. 信号值描述字段规则
---------------------
适用字段：
- signal_value_description

比较流程：
- 对两个值分别 normalize_description；
- 归一化结果完全相同则认为一致。

2.1 枚举格式解析规则
~~~~~~~~~~~~~~~~~~~~
如果文本能匹配枚举格式，则解析成 枚举值 -> 描述 的字典。

枚举 key 支持：
- 0x1
- 1
- 0x1~0x6
- 0x1-0x6
- 0x1`0x6
- 1~6
- 范围符号支持 ~、-、`、–、—
- 中文 ～ 会替换为 ~
- 中文 ｀ 会替换为 `

枚举格式要求：
- key 后面必须有冒号 : 或中文冒号 ：；
- 形如：0x1: Reserved。

2.2 枚举值归一规则
~~~~~~~~~~~~~~~~~~
- 文本转小写；
- 0x01 归一为 0x1；
- 0x0A 归一为 0xa；
- 十进制 10 会转成 0xa；
- 范围会展开为逐个 hex code；
- 如果范围起点大于终点，会按小到大展开。

2.3 枚举描述归一规则
~~~~~~~~~~~~~~~~~~~~
- 描述转小写；
- 描述中的 hex code 也归一；
- 删除所有空白；
- 删除描述末尾的 ; ； , ，。

2.4 枚举顺序规则
~~~~~~~~~~~~~~~~
- 枚举解析后按枚举值数值大小排序；
- 因此枚举项顺序不同但内容相同会认为一致。

可自动认为一致的例子：
- 0x01: ON 与 0x1: on
- 0x1: ON; 0x2: OFF 与 0x2: off; 0x1: on
- 0x1~0x3: Reserved 与 0x1:Reserved;0x2:Reserved;0x3:Reserved
- 1: ON 与 0x1: ON

风险：
- 只支持 key: desc 形式，ON=1 这类不一定能解析；
- 描述中如果出现类似 0x1: 的内容，可能被正则误切分；
- 范围太大时会展开大量 code；
- 十进制裸数字一定按十进制理解；
- 同义词、typo、翻译不支持。

2.5 非枚举描述归一规则
~~~~~~~~~~~~~~~~~~~~~~
如果无法解析成枚举：
- cell_text；
- 转小写；
- hex code 归一；
- 中文分号替换为英文分号；
- 中文冒号替换为英文冒号；
- 删除所有空白。

可自动认为一致的例子：
- Enable State 与 enable\nstate
- 0x01 value 与 0x1value

不支持：
- 同义词
- typo
- 中英文翻译
- 语义相同但表达不同
- 句子顺序变化

3. 普通文本字段规则
-------------------
适用字段：
- unit
- 以及数值字段无法转 Decimal 时的 fallback

比较流程：
- cell_text；
- 删除所有空白；
- 转小写；
- 完全相同则认为一致。

可自动认为一致的例子：
- km / h 与 KM/H
- V 与 v
- CAN FD 与 canfd

风险：
- 单位大小写如果有业务意义，会被忽略；
- m s 与 ms 会被认为一致；
- kph 与 km/h 不会自动认为一致；
- ℃ 与 degC 不会自动认为一致；
- % 与 percent 不会自动认为一致。


七、公共 ECU 状态解析工具规则
=============================

注意：部分公共函数当前主要服务于状态汇总/兼容解析，不一定全部参与最终文件五差异筛选。

1. parse_status_lines
---------------------
支持解析：
- EMS_PHEV:r
- HCU_PT:s
- TCU_PHEV:r(HighRegulationArea:r,LowRegulationArea:r)

解析规则：
- 先把 ( 替换为 ,，把 ) 删除；
- 再按换行、英文分号、中文分号、英文逗号、中文逗号拆分；
- 每段支持 ECU:state 或 ECU=state；
- state 如果后面还有多余内容，只取第一个字符。

风险：
- ECU 名中如果包含逗号、分号，可能被拆错；
- state 只取第一个字符，复杂状态会丢失后续内容。

2. aggregate_status_texts
-------------------------
聚合多条 ECU 状态文本：
- 可选择是否标准化状态；
- x 状态跳过；
- 同一个 ECU 若出现多个状态，用 / 连接；
- 保持 ECU 首次出现顺序。

3. split_send_receive_from_standard_status
------------------------------------------
从标准化状态文本中拆分发送 ECU 和接收 ECU：
- state 中包含 s -> 发送；
- state 中包含 r -> 接收；
- 支持 /、英文逗号、中文逗号拆分多个状态。


八、当前明确不支持或不参与的判断
===============================

1. 不判断新增 / 删除信号
------------------------
文件五只输出匹配成功且 7 个字段存在差异的信号。
只在 4.0 存在或只在 5.1 存在的信号，不作为新增/删除输出。

2. 不判断 ECU 收发变化
----------------------
ECU 字段只作为追溯字段带出，不参与差异筛选。

3. 不判断报文级变化
-------------------
以下字段不参与最终差异筛选：
- Msg ID
- Frame Format
- Msg Name
- Msg Length(Byte)
- Msg Send Type
- Transmitter

4. 不判断 Start Bit 和 Data Type 变化
------------------------------------
Start Bit 和 Data Type 在 01 全量文件中存在，但不进入 03 的 COMPARE_FIELDS。

5. 不做单位换算
---------------
不会自动判断：
- km/h 与 m/s
- ℃ 与 K
- rpm 与 r/min
- % 与 percent

6. 不做自然语言语义判断
-----------------------
不会自动判断：
- Enabled 与 ON 是否等价；
- Fault 与 Error 是否等价；
- Not Available 与 Unavailable 是否等价；
- typo 是否可忽略；
- 中英文翻译是否一致。

7. 不做通用模糊匹配
-------------------
除 VCU/HCU 去前缀匹配外，不支持通用信号名模糊匹配、别名表匹配、编辑距离匹配。


九、精确程度与误判风险总览
==========================

1. 精确程度高的规则
-------------------
- .xlsx / .xlsm 文件筛选；
- ~$ 临时文件排除；
- 表头前 40 行扫描；
- 核心字段缺失数量判断；
- R/r -> r；
- T/t/S/s -> s；
- X/x/× -> x 并跳过；
- Decimal 数值相等比较；
- 0x01 -> 0x1；
- 枚举项排序后比较；
- 4.0 ECU 括号格式输出；
- 5.1 ECU 逐行输出；
- 去重保留首次出现记录。

2. 精确程度中等的规则
---------------------
- 表头包含匹配；
- ECU 状态列 60% 阈值识别；
- 信号定义行至少 2 个证据字段；
- VCU/HCU 去前缀匹配；
- 枚举范围展开；
- 普通文本去空格、转小写比较；
- 单位字段比较。

3. 容易漏判的情况
-----------------
- 真实矩阵表头超出别名列表；
- 信号名大小写不同；
- 信号名中间空格、下划线、短横线不同；
- VCU/HCU 之外的命名前缀变化；
- 单位同义词或单位换算；
- 枚举描述同义词；
- typo；
- 新增/删除信号；
- ECU 收发变化；
- 报文级变化；
- Start Bit / Data Type 变化。

4. 可能误判为一致的情况
-----------------------
- 大小写在业务上有意义，但脚本忽略大小写；
- 空格在业务上有意义，但脚本删除空格；
- 单位中 m s 与 ms 被认为一致；
- 1 与 1.0 显示精度有业务意义，但脚本认为一致；
- 反向范围被容错展开；
- 十进制裸数字和 hex 表达的业务含义不一致。

5. 可能误判为不同的情况
-----------------------
- ON 与 Enabled 语义等价；
- Fault 与 Error 语义等价；
- kph 与 km/h 等价；
- ℃ 与 degC 等价；
- Reserved 与 RFU 等价；
- 英文和中文描述等价；
- 拼写错误但业务含义明显。

十一、AI辅助复核与人工审核明细规则（非 legacy 硬判断）
====================================================

本节描述 legacy 差异识别完成后的网页人工确认和最终审核结果导出规则；不改变前述四个 legacy 脚本的提取、去重和差异识别规则。

1. 人工确认粒度
---------------
人工确认按“信号 + 差异字段”保存，不再按整条信号保存五分类结论。同一信号的“信号值描述”和“单位”分别确认、分别更新，互不覆盖。

2. 人工确认结果
---------------
底层只保存两个稳定值：
- same：当前字段的 EEA4.0 与 EEA5.1 业务含义相同；
- different：当前字段的 EEA4.0 与 EEA5.1 业务含义不同。

界面按字段显示：
- 信号值描述相同 / 信号值描述不同；
- 单位相同 / 单位不同。

旧的“确认真实差异、确认可忽略、确认错别字、确认语义一致、存疑待确认”不再作为新任务的人工确认结果。

3. 系统判定
-----------
只对差异字段全部属于“信号值描述”和/或“单位”的信号调用 AI 并进入人工确认。只要同一信号包含任一数值类字段（信号长度、精度、偏移量、物理最小值、物理最大值），该信号的全部差异字段均直接记录为 different，判定来源为 system_default，不进入信号值描述或单位人工确认清单。

纯文本差异中的“信号值描述”和“单位”要求人工选择 same 或 different。人工审核先展示信号值描述清单；保存全部描述确认后，才展示单位清单。单位清单收录“仅单位”和“信号值描述+单位”两类信号；任务没有单位差异时不显示该清单。两个清单均完成后才允许生成最终结果。

审核页面在最终结果生成操作与结果文件下载之间展示“系统判定真实差异”只读表，按信号列出所有包含数值差异的信号及其数值字段差异。

4. 更新与并发
-------------
同一字段再次确认时更新原字段结果，不新增重复业务记录。每次更新继续执行任务审核锁校验、revision 乐观并发校验，并在 review_log.jsonl 中记录旧结果、新结果、字段标识、审核人和时间。

5. 最终审核 Excel
-----------------
文件名保持：人工审核后最终差异结果.xlsx。

工作表固定为：
- 人工确认不同：人工确认结果为 different 的字段；
- 人工确认相同：人工确认结果为 same 的字段；
- 系统判定不同：所有包含数值差异的信号，其全部差异字段均记录为 different；
- 待人工确认：尚未完成人工确认的字段；
- 审核明细全量：全部字段及判定来源、确认人、确认时间和稳定标识。

最终导出按字段一行，不再按旧五分类结果拆分工作表。存在待人工确认字段时，网页端禁止提交最终结果。
